# P2.4 — Recovery Engine + Orchestrator Intelligence Report

**Status:** ✅ Complete · Phase ratified contracts honored.
**Scope:** Recovery sovereign pillar + dynamic descriptor-tree orchestration.

## Deliverables

### Database (append-only + projection)
- `khalil_recovery_event` — append-only event log. UPDATE/DELETE blocked via `khalil_recovery_event_immutable` trigger. RLS owner-only.
- `khalil_recovery_state` — projection table (one row per user). Rebuilt by `khalil_recompute_recovery_state(user_id)` from the event log.
- `khalil_recovery_event_after_insert` trigger keeps projection in sync; idempotent (full recompute, last-event-wins).
- `khalil_adherence_daily` extended with `recovery_modifier` + `recovery_days_count` (softening hooks; never penalizes adherence).

### Domain (`src/core/khalil/recovery/`)
- `schemas.ts` — zod `recoveryModeSchema`, `toggleRecoveryInputSchema`, DTOs.
- `runtime/validateRecoveryTransition.ts` — pure state-machine. Allowed: `off↔soft`, `soft↔hard`. Forbidden: `off→hard` direct. `hard` requires reason ≥ 3 chars.
- `runtime/resolveRecoveryState.ts` — pure replay; mirrors SQL projection. Exposes `isRecoverySoftened`.
- `gateway/toggleRecovery.ts` — gated by `requireSupabaseAuth` + `requireKhalilCapability(RECOVERY_TOGGLE_WRITE)`. Reads current state, validates transition, inserts append-only event. Duplicate `client_event_id` → idempotent no-op.
- `gateway/readRecoveryState.ts` — gated by `RECOVERY_READ`. Returns state DTO + last 20 events for audit UX.

### Orchestrator (`src/core/khalil/orchestrator/`)
- `composeHome.ts` — descriptor-tree resolver. Inputs: recovery, time-of-day, capabilities, adherence summary. Pure TS.
  - `computeUrgencyScore(kind, ctx)` — deterministic urgency (recovery banner = -100, prayer promoted in Fajr / hard recovery).
  - `orderBlocksByUrgency(scored)` — filters invisible + sorts stable (urgency, then kind).
  - Hard recovery hides analytics depth + workout emphasis (guards in place for future pillars).
- `composeHome.functions.ts` — server fn now reads `khalil_recovery_state` + `khalil_adherence_daily` and defers to the pure resolver. No client logic.

### UI
- Block `khalil.recovery.banner` (`RecoveryBannerBlock`) — visible only when descriptor includes it. Read-only. Calm tone via `kt()` only.
- Page `KhalilRecoveryPage` at `/khalil/recovery` — current state, allowed transitions (filtered by `validateRecoveryTransition`), reason field for `hard`, offline-pending indicator, audit list (last 20 events).
- Block registered with `block.props.mode` plumbed through.

### Capabilities + Events
- `KHALIL_CAP.RECOVERY_TOGGLE_WRITE = "khalil.recovery.toggle.write"` (renamed from `RECOVERY_TOGGLE` to match contract).
- `KHALIL_CAP.RECOVERY_READ` unchanged.
- `KHALIL_EVENT.RECOVERY_CHANGED` already registered; emission happens via DB AFTER-INSERT trigger refreshing the projection.

- Conflict resolution: server timestamp + last-event-wins via projection recompute.
