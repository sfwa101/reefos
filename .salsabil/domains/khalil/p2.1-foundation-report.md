## Risk deviations

None requiring an ADR. Two intentional simplifications:

1. **`composeHome` returns serialized descriptor.** TanStack server-fn return
   inference replaces `unknown` with `{}` and rejects `Readonly<Record<string, unknown>>`.
   Mitigation: JSON-roundtrip + client-side cast to `RenderDescriptor`. No
   information loss, no runtime cost beyond a single network payload.
2. **`khalilOfflineQueue` is in-memory.** P1 mandates the *contract*; the
   IndexedDB-backed implementation lands with the first write capability in
   P2.2 (ADR-K003 — offline persistence).

## TODO boundaries for P2.2

P2.2 begins with **one** pillar (recommended: Prayer — smallest blast
radius, no recovery interaction). For that pillar P2.2 must:

1. Open ADR-K-P2.2-prayer with schema + RLS + capability-bundle migration.
2. Add `khalil_prayer_log` table + indices + RLS.
3. Register `khalil.prayer.log.write` / `read` capabilities server-side.
4. Implement `core/khalil/prayer/` runtime (pure TS) + gateway server fns.
5. Add `khalil.prayer.today` block + register in `blocks/register.tsx`.
6. Wire `KHALIL_EVENT.PRAYER_LOGGED` emission with idempotent subscriber.
7. Swap `khalilOfflineQueue` to IDB (`idb-keyval`) — same contract.
8. Add `requireCapability` server middleware (currently capability gating
   is namespace-only via constants; runtime check arrives with first
   write fn).

P2.2 is **forbidden** from:
- Touching the home block ordering rules until ≥2 pillars exist.
- Implementing identity-level math (deferred to P2.3 per ADR-K001).
- Wiring the AI coach (deferred to P2.4 per ADR-K005).
- Adding any cross-domain integration.
