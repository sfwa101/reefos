# P3.1 — Sovereign Intelligence Layer Report

**Status:** complete.
**Scope:** foundational adaptive intelligence runtime. No chat, no LLM, no client inference.

## Architecture

```
src/core/khalil/intelligence/
├── contracts/types.ts        SovereignSignal, NudgeProposal, WeeklyFocusPlan,
│                             PriorityScore, IntelligenceInputs, IntelligenceSnapshot
├── signals/computeSignals.ts 10 deterministic signal computations

├── nudges/composeNudges.ts   Auditable, source-cited nudges (max 3)
├── planning/composeWeeklyPlan.ts  Structured weekly focus

├── tests/intelligence.test.ts  Determinism + scenario tests
└── index.ts                  Barrel (UI imports via @/core/khalil)

Caps: khalil.intelligence.read, khalil.intelligence.recompute
```

## Replay guarantees

- All engines are PURE. No `Date.now()`, no `Math.random()`, no `fetch`, no I/O.
- Single time source: server captures `now` once and threads it through every record
  via `IntelligenceInputs.now`. Same inputs → byte-identical snapshot.
- Nudge ids are FNV-1a derived from `(kind, generatedAt)` — no UUIDs.
- `inputsDigest` (FNV-1a over canonical, fixed-precision serialization) lets the
  replay runner detect drift between live snapshots and replayed history.
- Append-only table: replay re-inserts; older snapshots are never mutated.

## Scoring philosophy

Static base weights per pillar (`BASE_WEIGHT`) are overlaid by signal-driven
adjustments. Lower weight = earlier in `composeKhalilHome`. The orchestrator
combines:

1. Base urgency (existing `computeUrgencyScore`)
2. Intelligence priority overlay (authoritative when present)

Examples:
- Hard recovery → `khalil.workout.next` weight += 200, banner stays −100.
- `prayer_streak_risk ≥ 60` → prayer block −40.
- `momentum_gain ≥ 60` → analytics surfaces −15.
- Coach pending + identity ≥ disciplined → proposal surfaced above welcome.

No ML, no black box — every adjustment is a named numeric constant.

## Determinism proof

`composeIntelligenceSnapshot(inputs)` is the composition of four pure functions
over a frozen `IntelligenceInputs`. Tests (`tests/intelligence.test.ts`) lock:

- two equal inputs → identical JSON snapshot,
- nudge ids stable across runs,
- 10 signal keys always present and bounded 0..100,
- hard recovery forbids body overload + ranks recovery banner first,
- falling prayer adherence raises `prayer_streak_risk ≥ 60`.

## Governance constraints

`scripts/khalil-intelligence-governance.mjs` blocks:

- `Math.random`, `Date.now()`, `new Date()`, `crypto.randomUUID` in engine dirs
- Direct `fetch(`, openai/anthropic/@ai-sdk imports in intelligence layer
- UI files importing `@/core/khalil/intelligence/{signals,scoring,nudges,planning,replay}`
  (UI must go through `@/core/khalil` barrel + gateway fns)

Combined with the existing `scripts/khalil-governance.mjs`, the intelligence layer
inherits: no cross-domain imports, no direct Supabase in UI, no hardcoded Arabic
in TSX. All nudge/signal copy is routed through `kt()`.

## Risk analysis

| Risk | Mitigation |
|---|---|
| Snapshot drift over time | `inputsDigest` + replay runner integration; append-only history |
| Manipulative copy | Nudge kinds are a closed enum; copy lives in i18n; no fear language reviewed in `khalil.intelligence.nudge.*` keys |
| Client-side inference creep | Governance script blocks UI engine imports; blocks render descriptor props only |
| Non-determinism from clock | Single `now` threaded through every record; engines reject `Date.now` via tripwire |
| Capability bypass | All gateway fns wrap `requireKhalilCapability(INTELLIGENCE_*)` |
| Excess nudging | Hard cap: `MAX_NUDGES = 3`; nudges expire 12h after `generatedAt` |

## Extension points for P3.2

- Add new signal: append to `SOVEREIGN_SIGNAL_KEYS`, implement in `computeSignals`,
  add `khalil.intelligence.signal.<key>.{title,explanation}` i18n keys, bump
  `INTELLIGENCE_REPLAY_VERSION` so replay can branch on version.
- Add nudge kind: extend `NUDGE_KINDS`, add mapping rule in `composeNudges`, ship
  `khalil.intelligence.nudge.<kind>.{title,body}` keys.
- AI provider integration (P3.2+): MUST land as a separate server gateway that
  produces typed candidates the dispose ritual already supports (Art. XI). The
  intelligence snapshot remains the deterministic baseline; AI never writes here.
- Per-user signal weighting: introduce `intelligence_profile` projection; engines
  stay pure (weights injected via inputs).
