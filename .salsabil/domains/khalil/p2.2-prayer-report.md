## 5. Event Flow

```
UI press
  → logPrayerFn (server fn, RLS as user)
    → validatePrayerWindow (pure)
    → INSERT khalil_prayer_log
        ├─ unique(user,date,prayer,mode)   → semantic dedupe
        └─ unique(client_event_id)         → replay dedupe
    → AFTER INSERT trigger
        → khalil_recompute_adherence(user, date)
            → UPSERT khalil_adherence_daily
```

Event envelope (`khalil.prayer.logged`, v1) is documented in
`src/core/khalil/prayer/events.ts`. P2.2 surfaces it implicitly via the
DB projection trigger; the application-layer subscriber lands when the
sovereign event bus is online (see ADR-K00x backlog).

## 6. Replay Verification

- `khalil_recompute_adherence` performs a **full recompute** of one day
  from the log → calling it N times on the same day is identical to
  calling it once. ✅ idempotent.
- `replayPrayerProjectionFn` enumerates distinct logged days for the
  caller in `[fromDate, toDate]` and replays each. Output:
  `{ replayed: <count>, days: [...] }`.
- Manual verification path:
  1. Insert prayer rows for a past week (via gateway).
  2. `DELETE FROM khalil_adherence_daily WHERE user_id = $me`.
  3. Call `replayPrayerProjectionFn({ fromDate, toDate })`.
  4. `SELECT * FROM khalil_adherence_daily` → rows reconstructed.

## 7. Offline Verification

- Queue persistence: `idb-keyval` under `khalil.offlineQueue.v1`.
  Survives reloads; falls back to in-memory only when IndexedDB is
  unavailable (SSR, locked-down browsers).
- UI path: `navigator.onLine === false` → enqueue with
  `idempotencyKey = clientEventId`; optimistic UI shows pending pill
  (`khalil.prayer.offline.pending`).
- Server idempotency: replay of the same `clientEventId` returns
  `{ status: "duplicate", kind: "replay_duplicate" }` instead of
  surfacing an error. Semantic duplicates (same user/date/prayer/mode
  with a fresh `clientEventId`) collapse to `kind: "semantic_duplicate"`.
- Replay drain wiring (the queue → gateway loop) intentionally stays
  out of P2.2 — `khalilOfflineQueue.peek()` is enough for the block's
  pending pill; the executor lands with the network-status integration
  ADR in P2.3.

## 9. Known Limitations (Tracked, NOT Deviations)

1. **Coarse prayer windows.** Hour-band check, not astronomical. Real
   sun-calc requires a calculation library and per-user geo. Owned by
   ADR-K-PRAY-01 (backlog).
2. **DB-trigger projection.** P1 contract allows trigger-as-subscriber
   while the sovereign event bus is offline; full bus subscriber lands
   with the AI coach phase.
3. **No queue drainer.** Offline writes accumulate but aren't auto-
   flushed — block surfaces the pending pill so the user sees the
   state. Drainer is its own ADR (P2.3).
4. **No qadaa overwrite path.** A second `qadaa` log for the same
   (user, date, prayer) is silently classified `semantic_duplicate`.
   Spec said "cannot overwrite existing entry" — current behaviour
   matches; explicit user feedback ("already logged") is a UX polish
   ticket, not an architectural fix.
