# Phase P0.1 — Coupling Risk Analysis

Hidden couplings detected during P0.1 audit. Each is rated by **severity**
and **directionality** (which domain leaks into which).

## 1. SDUI slug `khalil_hub` is still wired to Maeen content

- **Severity:** Medium.
- **Direction:** Data (DB) → Maeen runtime.
- **Detail:** `sdui_layouts.slug = 'khalil_hub'` is queried by
  `MaeenLauncherGrid` (`const SLUG = "khalil_hub"`) and by
  `HakimGenerativeOverlay.injectBlock(SLUG, ...)`. The name is a
  historical artifact from when Maeen lived under `src/apps/khalil/`.
- **Risk:** A future engineer reading the DB row will assume it belongs
  to Khalil (the transformation OS) and may attempt to repurpose it,
  breaking the Maeen hub.
- **Mitigation in P0.1:** Code-side comments at both call sites point
  to this report. The constant is named `SLUG` (not `KHALIL_SLUG`) so
  the value is the only legacy surface left.
- **Resolution path:** **ADR-0004** — schema migration renaming the row
  to `maeen_hub`, dual-read window, then drop the alias.

## 2. `OS_MODULES.shared_by` carried Maeen consumers under `khalil`

- **Severity:** Low (resolved in this phase).
- **Direction:** Identity registry → capability resolution (future).
- **Detail:** Before P0.1, `wallet` and `catalog` listed `khalil` as a
  consumer because the `khalil` entry actually meant Maeen. Any future
  resolver iterating `shared_by` would have granted the transformation
  OS modules it has no legitimate need for.
- **Resolution:** `osCompanies.ts` rebalanced — see ADR-0003 §5.
- **Standing risk:** The shape of `OS_MODULES` (a flat module registry
  declared as data) is itself a soft governance surface. It needs to
  become driven by **capability declarations** under `src/core/capabilities/`
  rather than hand-maintained string arrays. Tracked as a Capability
  follow-up (out of P0.1 scope).

## 3. `LayoutFactory` directly imports civilization components

- **Severity:** Medium (pre-existing; surfaced by P0.1 audit).
- **Direction:** Reef storefront → Maeen civilization.
- **Detail:** `LayoutFactory.tsx` statically imports `MaeenLauncherGrid`
  from `@/apps/maeen/...`. The factory lives inside `src/apps/reef-al-madina/`
  yet hard-codes knowledge of a sibling civilization. This is the same
  cross-civilization coupling that ADR-0003 just removed at the
  Khalil↔Maeen seam — it now manifests as Reef↔Maeen.
- **Why not fixed now:** Out of scope. P0.1 is forbidden from
  introducing new runtime systems. Fixing this requires registering
  Maeen blocks via the SDUI runtime-ui registry and removing the
  static import — a runtime-ui task with its own ADR.
- **Resolution path:** ADR (Maeen-side) — `MaeenLauncherGrid` registered
  via `@/core/runtime-ui/sdui/registry.ts` and lazily resolved by
  `LayoutFactory` instead of statically imported.

## 4. `useActiveDelivery` lives under `src/core/maeen/`

- **Severity:** Low.
- **Direction:** Maeen presentation → a `core/maeen/` module that does
  not match the project's domain ownership map (no `src/core/maeen/`
  entry in `.salsabil/architecture/02-ownership-map.md`).
- **Why it matters:** Either Maeen earns a sovereign `src/core/maeen/`
  gateway in the ownership map, **or** `useActiveDelivery` belongs in
  `src/core/logistics/` (it reads delivery state). Today it's neither
  fish nor fowl.
- **Resolution path:** ADR-M001 (Maeen domain memory) will pick a side
  and the ownership map will be updated accordingly.

## 5. `OSCompanyId` doubles as identity AND switcher routing key

- **Severity:** Informational.
- **Detail:** `APP_ROUTES: Record<OSCompanyId, string>` couples the
  identity registry (`osCompanies.ts`) to navigation routing. Today
  this is intentional and useful (compile-time exhaustiveness check
  surfaced the missing `maeen` route immediately when the union was
  extended). Becomes a problem only if non-routable identities are
  added; revisit then.

## Net assessment

P0.1 closes the two **hard** couplings (directory mis-ownership and
`OS_MODULES` mis-attribution) and **documents** three pre-existing
couplings that exceed its scope. No new coupling was introduced.
