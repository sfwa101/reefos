# P2.6 — Sovereign AI Coach (Proposal/Dispose) — Implementation Report

Status: **Shipped within contract.** All P1/P2 invariants intact. No ADR deviations.

## 1. Domain model
- `khalil_coach_proposal` (append-only lifecycle, immutable fields, terminal status guard, no DELETE).
- `khalil_coach_audit` (append-only; `proposed | accepted | dismissed | expired | rejected`).
- RLS: owner-scoped on both tables. Triggers `khalil_coach_proposal_lifecycle_guard`, `khalil_coach_proposal_no_delete`, `khalil_coach_audit_immutable` enforce invariants in the DB.

## 2. Runtime (`src/core/khalil/coach/`)
- `schemas.ts` — Zod schemas with `.strict()`, capability allowlist (`RECOVERY_TOGGLE_WRITE`, `HABIT_COMPLETE_WRITE`, `PRAYER_LOG_WRITE` only). No free-form text fields; only i18n keys.
- `runtime/schema.ts` — single sanctioned validator (`validateCoachProposalDraft`).
- `runtime/prompts.server.ts` — server-only prompt registry, `PROMPT_VERSION = khalil.coach.v1`, deterministic snapshot projection, allowlisted copy keys (`ALLOWED_COPY_KEYS`). Filename suffix prevents client bundling.
- `runtime/snapshot.ts` — allowlisted shape (identity level, recovery, adherence, active pillars, recent proposal kinds). No raw chat, no journaling, no cross-domain data.
- `runtime/fallback.ts` — deterministic quiet-day fallback (§11).

## 3. Gateways
- `proposeCoachFn` (`khalil.coach.propose.read`) — builds server snapshot → prompt → AI gateway (`ai.gateway.lovable.dev`) → strict validation → `withSafeCopyKey` belt-and-suspenders → persist append-only → audit `proposed` (`source: model | fallback`).
- `acceptCoachFn` (`khalil.coach.accept`) — re-validates ownership, status, expiry. Idempotent on terminal status. Emits `expired` audit when crossed during accept. Never writes domain rows; surfaces `suggestedCapability` for the user to execute on the relevant page through its own capability-gated gateway.
- `dismissCoachFn` (`khalil.coach.dismiss`) — idempotent lifecycle update + audit.
- `readCoachHistoryFn` — current pending + last 20 terminal proposals.

## 4. Events
- Added `KHALIL_EVENT.COACH_EXPIRED`. Existing `COACH_PROPOSED/ACCEPTED/DISMISSED` retained. Emission is replay-safe via the append-only audit table (subscribers read the projection, never the trigger payload).

## 5. Orchestrator
- `composeKhalilHome` accepts optional `pendingCoachProposal`; emits a single `khalil.coach.proposal` block when present.
- `computeUrgencyScore`: quiet placement (-20), softened to +25 for `disciplined+`, +40 in hard recovery.
- `visibilityFor`: hard recovery restricts to `recovery-suggestion | quiet-day`; sovereign restricts to `quiet-day | consistency-guidance`. Max one visible proposal.
- `composeKhalilHomeFn` fetches the latest pending row (status=pending, not expired) and feeds id+kind only.

## 6. UI
- `/khalil/coach` (`KhalilCoachPage`) — pending proposal card + refresh + 20-entry history. No textarea, no chat, no free-form prompting.
- `CoachProposalBlock` — single proposal with accept/dismiss; offline-queued via `khalilOfflineQueue` with `COACH_ACCEPT` / `COACH_DISMISS` capabilities; pending-sync chip; idempotent replay (server rejects expired/terminal, no-ops on duplicate).
- All copy resolved through `kt()`; new keys cover proposal copy/body, kind labels, status labels, action labels, offline label.
- Block registered in `register.tsx` for descriptor-driven home composition.

## 7. Safety invariants (verified)
- Coach never bypasses capability checks (both write paths re-gated server-side).
- Coach never writes Supabase domain rows directly — only its own proposal/audit lifecycle.
- Coach never stores or renders raw provider output — strict schema + copy-key allowlist + fallback.
- Prompts live in `*.server.ts` and are imported only from server-fn gateways.
- Proposals are time-bound (`ttlSeconds`, hard cap 24h; expiry checked on accept).
- Provider/model identity hidden from UI; UI sees DTOs only.

## 8. Tests

- `runtime/__tests__/prompts.test.ts` — determinism (identical snapshots → identical prompts), pillar-ordering normalization, prompt version stable.
