# P2.7 — Body + Analytics Completion Report

## Schema inventory
- `khalil_workout_session` — open/close lifecycle. Partial unique index enforces single open session per user. Immutability trigger forbids mutation of closed sessions and of immutable fields. No DELETE policy.
- `khalil_workout_set` — append-only sets. RLS: SELECT + INSERT only. No UPDATE/DELETE policies; corrections are new rows with `is_correction = true` referencing the original via `corrects_set_id`.
- `khalil_workout_projection_weekly` — (user_id, iso_year, iso_week). Maintained by triggers on session insert + set insert.
- `khalil_weight_measurement` — one measurement per `(user_id, for_date)`. Plausible kg bounds (20–500) enforced via CHECK. Append-only RLS.
- `khalil_weight_projection` — latest/avg/delta rollups. Trigger-maintained on every measurement insert.
- `khalil_streak_state` — per-user current/longest streaks for prayer/habit/combined. Recomputed via trigger on `khalil_adherence_daily` AFTER INSERT/UPDATE.
- `khalil_analytics_event` — append-only audit log surface for future analytics jobs.

## Capability inventory (re-used; no new keys needed)
- `khalil.workout.session.write`, `khalil.workout.set.write`, `khalil.workout.read`
- `khalil.weight.measurement.write`, `khalil.weight.read`

## Event inventory
- `khalil.workout.session_opened`, `khalil.workout.session_closed`, `khalil.workout.set_recorded`
- `khalil.weight.recorded`

## Projection topology
- Writes flow through gateways → DB triggers maintain projections deterministically.
- Application-bus subscribers (`workoutProjectionSubscriber`, `weightProjectionSubscriber`) are idempotent on `event.id` and serve as future fan-out seams; persistence remains DB-trigger-owned per Art. IV §B.
- Read gateways consume PROJECTIONS only (`khalil_adherence_daily`, `khalil_weight_projection`, `khalil_workout_projection_weekly`).

## Replay guarantees
- `replayWorkoutWeekly()` deterministically rebuilds weekly volume from sessions + sets (drops superseded sets via `effectiveSets`).
- `replayWeightTrend()` mirrors the DB trigger logic; pure TS, no I/O.
- Stale-session auto-close uses pure `isSessionStale()` predicate — replay and live read both deterministic.

## Offline guarantees
- Writes for weight + workout flow through `khalilOfflineQueue` (IndexedDB) when `navigator.onLine === false`.
- Server idempotency on `(user_id, client_event_id)` — duplicate replays are no-op successes.
- FIFO ordering preserved by queue contract.

## Mobile bundle impact
- Home route bundle: NO chart libraries. Heatmap renders via plain CSS grid cells; adherence renders as plain CSS bars.
- Insights route: lazy-loads `InsightsCharts.tsx` via `React.lazy`. Charts (CSS bars, no third-party lib) only pulled when the user lands on `/khalil/insights`.

## Orchestrator integration
- `composeKhalilHome` now emits four new candidate blocks: `khalil.workout.next`, `khalil.weight.trend`, `khalil.analytics.adherence`, `khalil.analytics.heatmap`.
- Urgency rules:
  - Workout: hidden in hard recovery, softened in soft recovery, promoted for rising+ identities.
  - Weight: demoted in hard recovery, promoted for stable+ identities.
  - Analytics: hidden for `seed` and `hard` recovery; promoted for `disciplined`+.
- All ordering server-side; client renders descriptor only.
