— ديباجة الميثاق —

سلسبيل ليست تطبيقاً. سلسبيل نظام تشغيل حضاري سيادي، يهدف إلى بناء طبقة اقتصادية واجتماعية وتعليمية وصحية ومالية موحدة، تخدم الإنسان المسلم وتحترم شريعته وكرامته وخصوصيته.

ريف المدينة ليس متجراً إلكترونياً. هو دولة اقتصادية رقمية لها مؤسساتها وصلاحياتها ومناطقها وسياساتها وعملاتها الداخلية وشركاؤها. وكل سطر كود يُكتب في هذا المشروع يجب أن يكون واعياً بهذه الرؤية.

هذا الدستور النسخة الثانية الموحدة — يجمع المبادئ المعمارية الأصلية مع خريطة المحركات الكاملة (١٦ محركاً حضارياً + ١٤ محركاً تجارياً) ونظام شركاء النجاح وبنية الخلايا الجذعية. لا مكان للحلول المؤقتة، ولا للمنطق المُرمَّز، ولا للتفكير بعقلية المتجر.

الباب الأول

الفلسفة التأسيسية والمبادئ العليا

المادة ١.١

سلسبيل كيان سيادي لا تطبيق

يُبنى كل مكون في المشروع بعقلية "نظام دولة"، لا بعقلية "صفحة ويب". كل قرار معماري يجب أن يصمد أمام سؤال: هل هذا القرار يخدم منصة تخدم مليار إنسان؟ ريف المدينة ليس متجراً — هو دولة اقتصادية مصغرة لها مؤسساتها وسياساتها.

المادة ١.٢

الشريعة الإسلامية دستور علوي

الشريعة الإسلامية المرجعية العليا في كل قرار — في الخصوصية، والفصل بين الجنسين، والمعاملات المالية، وأخلاقيات البيانات، وعدالة الوصول. تُترجم الشريعة إلى Policy Engine وليس إلى قيود ترميز ثابتة.

المادة ١.٣

الإنسان أولاً — البيانات خادمة

البيانات أمانة، لا سلعة ولا سلاح مراقبة. لا يُباع بيانات العملاء. لا يُستغل البيانات الطبية. الخصوصية حق أصيل يسبق أي اعتبار تجاري. البيانات ملك سيادي للكيانات والبشر — لا للشركة.

المادة ١.٤

الذكاء الاصطناعي مستشار لا سيد

حكيم يقترح ويحلل ويحذر — لكنه لا يقرر سيادياً أبداً. القرار النهائي للإنسان دائماً. الذكاء الاصطناعي لا يملك صلاحية تعديل السياسات أو الثيمات أو الصلاحيات أو العقود تلقائياً دون موافقة بشرية صريحة.

المادة ١.٥

Civilization As Runtime — الحضارة كنظام تشغيل

الهدف النهائي ليس بناء تطبيقات منفصلة — بل بناء نواة تشغيل حضارية واحدة يعمل فوقها ريف المدينة، نبض، تيسير، نور الدين، برق، وأي كيان مستقبلي. كل كيان يضيف قدرات (Capabilities) فوق نفس النواة — لا يُعاد بناء النواة من الصفر.

المادة ١.٦

ريف المدينة يبيع أولاً

القاعدة الذهبية للأولويات: ريف المدينة يجب أن يُصدر فاتورة حقيقية أولاً قبل بناء أي محرك حضاري. لا تُبنى features متقدمة قبل وجود حركة تجارية حقيقية. الرؤية العملاقة تُبنى طبقة فوق طبقة — كل طبقة تعمل وحدها وتولد قيمة حقيقية.

الباب الثاني

المحظورات المطلقة — ABSOLUTE FORBIDDEN

⚠ تحذير سيادي — أي انتهاك يُوقف العمل فوراً

هذه القوانين غير قابلة للتفاوض. لا يوجد سبب كافٍ لانتهاكها — لا الوقت، ولا الضغط، ولا "الحل المؤقت". كل Hack اليوم يتحول إلى كارثة بعد سنة.

المحظورات المعمارية المطلقة

  • أي استدعاء مباشر لقاعدة البيانات من داخل مكونات الواجهة (UI Components)
  • أي supabase.from("table") خارج طبقة الـ Gateway والـ Services
  • أي منطق تجاري (Business Logic) داخل مكونات React أو الصفحات
  • أي if (section === "restaurant") أو تفريع بالاسم في أي مكان
  • أي قيم مرمّزة Hardcoded — أسماء أقسام، معرّفات، سياسات — في الكود
  • أي تكرار لنفس الـ Hook أو الـ Component بتعديل طفيف لكل حالة
  • أي تجاوز للـ Runtime Renderer ببناء صفحة مخصصة لكيان بعينه
  • أي استخدام لـ localStorage أو sessionStorage لبيانات حساسة أو صلاحيات
  • أي God Mode أو bypass في بيئة الإنتاج Production
  • أي تعديل مباشر على سجلات محاسبية أو مالية (UPDATE balance = balance - x)
  • أي وصول لبيانات tenant آخر تحت أي ظرف
  • أي key خاص (service role, AI key) في بيئات الـ frontend أو الـ logs
  • بناء صفحة React جديدة لكل نوع نشاط — بدلاً من seed + capabilities
  • أي AI يعمل باستقلالية كاملة دون Human Veto في الإنتاج
  • نظام عمولات مرتبط بالتجنيد لا بالخدمة الحقيقية (MLM محظور)

قاعدة الاختبار الذهبية: إذا كنت تفكر "سأضع هذا مؤقتاً وأصلحه لاحقاً" — هذا الإحساس بالضبط هو إشارة التوقف. لا توجد "لاحقاً" في المشاريع الضخمة.

الباب الثالث

الواجبات الإلزامية — ABSOLUTE REQUIRED

الواجبات المعمارية الإلزامية

  • كل قراءة للبيانات تمر عبر catalogGateway أو entityService المختص
  • كل واجهة تستقبل فقط ViewModel جاهز — لا raw data من DB
  • كل قسم يُعرَّف عبر capabilities وpolicies — لا صفحة خاصة
  • كل block في الواجهة مسجّل في blockRegistry بشروط ظهور capability-driven
  • كل عملية مالية تُسجَّل كـ immutable ledger event — لا تعديل مباشر
  • كل طلب يحمل tenant_id موثّقاً من السيرفر فقط
  • كل عملية حساسة تملك idempotency_key لمنع التكرار
  • كل تغيير جوهري يُسجَّل في audit_events غير قابل للتعديل
  • كل capability جديد يُسجَّل في capabilityRegistry قبل الاستخدام
  • كل نوع بيانات يُعرَّف في TypeScript interface مركزي موحّد
  • كل حدث في النظام يُنشر عبر EventBus — لا مزامنة مباشرة بين المحركات
  • كل منتج يمتلك Product DNA كاملاً — لا مجرد اسم وسعر
  • كل شريك نجاح تربط عمولته بـنجاح العميل لا بعدد التجنيد
  • كل output من AI يمر عبر Human Veto قبل التنفيذ

الباب الرابع

معمارية الطبقات — Sovereign Layered OS

المادة ٤.١

الطبقات السبع — لا تخلط بينها أبداً

المشروع مبني على سبع طبقات مفصولة تماماً. كل طبقة لا تتواصل مع ما تحتها مباشرة — بل عبر الطبقة التي تليها فقط.

Layer 7 — UI

مكونات React فقط. تستقبل ViewModels. لا تعرف DB. لا تعرف Supabase.

Layer 6 — Runtime

RuntimeRenderer + BlockRegistry + FeedEngine. يولّد الواجهة من descriptors.

Layer 5 — Application

UseCases + ViewModels + Transformers. Business logic هنا فقط.

Layer 4 — Domain

Entities + Capabilities + Policies + Workflows + Product DNA.

Layer 3 — Gateway

CatalogGateway + EntityService + FinanceGateway. الوسيط الوحيد مع DB.

Layer 2 — Infrastructure

Supabase + Storage + Queue + Cache. Implementation details فقط.

Layer 1 — Events

Event Bus (NATS/Supabase Realtime). كل تغيير مهم ينشر event. لا coupling مباشر.

قانون الطبقات المطلق

Layer 7 (UI) لا تعرف وجود Supabase. Layer 6 (Runtime) لا تعرف أسماء الجداول. Layer 5 (Application) لا تعرف طريقة التخزين. Layer 1 (Events) هو الجهاز العصبي — كل حدث ينتشر عبره لا مباشرة.

الباب الخامس

نظام الكيانات والقدرات — Entity & Capability System

المادة ٥.١

كل شيء كيان — Everything is an Entity

لا يوجد "متجر" أو "مطعم" أو "صيدلية" كصفحات ثابتة. كل هذه مجرد Entities بـ capabilities مختلفة. الـ Runtime يقرر كيف يعرضها. حتى المنتج نفسه كيان حي — له DNA، وذاكرة، وسلوك، ودورة حياة.

// كل كيان له هذا الهيكل الأساسي interface Entity { entity_id: "UUID"; entity_type: "Human" | "Store" | "Product" | "Ministry"; state: "Active" | "Draft" | "Suspended"; context: Record<string, unknown>; relationships: Relationship[]; capabilities: Capability[]; memory: EntityMemory; events: Event[]; policies: Policy[]; } // ❌ خاطئ — تفريع بالاسم if (sectionType === 'restaurant') return <RestaurantPage />; // ✓ صحيح — capability-driven const caps = capabilityRegistry.resolve(section.id); const descriptor = buildRenderDescriptor(caps); return <RuntimeRenderer descriptor={descriptor} />;
المادة ٥.٢

Capability Registry — سجل القدرات الموحد

كل capability اسم منطقي يُمكّن سلوكاً معيناً. لا يُضاف capability جديد بدون تسجيله هنا أولاً.

export const CAPABILITIES = { // تجاري SUPPORTS_VARIANTS: 'supports_variants', SUPPORTS_ADDONS: 'supports_addons', SUPPORTS_NUTRITION: 'supports_nutrition', SUPPORTS_WHOLESALE: 'supports_wholesale', SUPPORTS_SUBSCRIPTION: 'supports_subscription', SUPPORTS_COLD_CHAIN: 'supports_cold_chain', SUPPORTS_KITCHEN_MODE: 'supports_kitchen_mode', SUPPORTS_QUICK_BUY: 'supports_quick_buy', SUPPORTS_PRESCRIPTION: 'supports_prescription', // مالي SUPPORTS_LEDGER: 'supports_ledger', SUPPORTS_DEFERRED_PAY: 'supports_deferred_pay', SUPPORTS_COMMISSION: 'supports_commission', SUPPORTS_OFFLINE_POS: 'supports_offline_pos', // ذكاء SUPPORTS_AI_PRICING: 'supports_ai_pricing', SUPPORTS_VISION: 'supports_vision', SUPPORTS_PRODUCT_DNA: 'supports_product_dna', SUPPORTS_AGENTS: 'supports_agents', };
الباب السادس

نظام الصلاحيات — Sovereign Permission Model

المادة ٦.١

الصلاحيات سياقية متعددة الأبعاد

الصلاحية لا تُعرَّف بـ role بسيط. تُعرَّف بـ ٤ أبعاد: من أنت + في أي كيان + في أي نطاق + ماذا تفعل.

// ❌ خاطئ if (user.role === 'admin') { showInventory(); } // ✓ صحيح — multi-dimensional const canView = permissionEngine.check({ actor: user.id, action: 'inventory.view', resource: branchId, scope: { region: 'egypt', entity: 'reef-madina' } });

قواعد الصلاحيات الإلزامية

  • لا يُقرأ tenant_id من الـ client أو الـ route أو الـ URL — السيرفر فقط
  • لا تُمنح صلاحية Admin عامة — كل صلاحية مقيّدة بنطاق محدد
  • لا يرى مستخدم بيانات خارج نطاق صلاحيته مطلقاً
  • لا تُخزَّن الصلاحيات في localStorage أو حالة الـ client
  • كل escalation صلاحية يُسجَّل في audit log فوراً
  • شريك النجاح لا يرى بيانات العملاء الشخصية — فقط مؤشرات الأداء
الباب السابع

المعمارية المالية — Immutable Financial Architecture

المادة ٧.١

كل عملية مالية حدث لا يُحذف

لا يوجد UPDATE أو DELETE على سجلات مالية. كل عملية Append-Only Event. الرصيد الحالي مشتق من مجموع الأحداث — لا يُخزَّن مباشرة. ينطبق هذا على: محفظة العميل، عمولة الشريك، فاتورة المورد، كل شيء مالي.

// ❌ ممنوع مطلقاً UPDATE wallet SET balance = balance - 150 WHERE user_id = $1 // ✓ الطريقة الوحيدة المقبولة INSERT INTO ledger_events ( id, wallet_id, type, amount, idempotency_key, reference, created_at ) VALUES ( gen_random_uuid(), $wallet_id, 'debit', 150, $idempotency_key, $order_id, now() ); -- الرصيد يُحسب دائماً من السجل SELECT SUM(CASE type WHEN 'credit' THEN amount ELSE -amount END) FROM ledger_events WHERE wallet_id = $1;

قواعد المالية الإلزامية

  • كل عملية دفع تملك idempotency_key فريداً لمنع الخصم المزدوج
  • كل تحويل يستخدم double-entry ledger (مدين وائتمان معاً)
  • كل عملية مالية تُلاحَق بـ trace_id طوال دورة حياتها
  • عمولات شركاء النجاح تُحسب من نفس Ledger Engine — لا نظام منفصل
  • لا يُنفَّذ دفع بدون تحقق من الرصيد على السيرفر — لا على الـ client
الباب الثامن

قانون الذكاء الاصطناعي — حكيم في حدوده

Advisory — مستشار

يقترح، يحلل، يحذر. لا ينفذ. المستخدم يقرر.

Assisted — مساعد

يُنفّذ بعد موافقة صريحة Human Veto. كل تنفيذ يُسجَّل.

Autonomous — محظور

لا AI يعمل باستقلالية كاملة في الإنتاج أبداً.

ما يُمنع على حكيم مطلقاً

  • تعديل ثيمات أو هوية بصرية تلقائياً
  • تغيير صلاحيات مستخدمين بدون موافقة صريحة
  • تنفيذ عمليات مالية أو عمولات بدون تأكيد بشري
  • تعديل سياسات النظام أو قوانين الـ runtime
  • الوصول لبيانات المستخدمين الشخصية خارج سياق الطلب
  • تنفيذ أوامر حذف على بيانات حقيقية
  • تعديل هيكل قاعدة البيانات أو الـ Schema

Pipeline إلزامي لكل output من AI

  • Schema Validation — هل الشكل صحيح؟
  • Business Validation — هل المنطق مقبول؟
  • Permission Validation — هل مسموح؟
  • Rate Limiting — هل ضمن الحدود؟
  • Human Veto — هل وافق الإنسان؟
  • Execution + Audit Log — التنفيذ مع التوثيق الكامل
الباب التاسع

قواعد البناء بـ Lovable

المادة ٩.١

Lovable مصنع — لا مهندس معمارية

Lovable ينفّذ ما يُملى عليه بدقة. المعمارية قرار بشري يُملى عليه. مهمته: بناء الـ blocks والـ components والـ services وفق الدستور فقط.

هيكل كل Prompt لـ Lovable — إلزامي

  • السياق المعماري: في أي Layer نعمل؟ ما موقعه في النظام؟
  • الملفات المحددة: ماذا يلمس بالضبط وماذا لا يلمس؟
  • الممنوعات الصريحة: اذكر بالاسم ما لا يفعله
  • معايير القبول: كيف تعرف أن الناتج صحيح؟
  • حجم المهمة: لا تعطِ مهمة تستغرق أكثر من ٣٠ دقيقة مراجعة
""" السياق: نحن في Layer 5 (Application). نبني ProductViewModelFactory. المهمة: أنشئ ملف src/core/catalog/factories/ProductViewModelFactory.ts يحول ProductEntity → ProductCardVM. الممنوع: - لا تستدعِ Supabase مباشرة - لا تضع business logic خارج هذا الملف - لا تعدّل أي ملف آخر - لا any types معايير القبول: - ProductCardVM يحتوي: id, title, price, thumbnail, capabilities[], dna - الدالة تقبل ProductEntity[] وتُرجع ProductCardVM[] - TypeScript strict mode """
قانون الـ Micro-Prompt

المهام الكبيرة تُقسَّم إلى مهام صغيرة متتالية. كل مهمة تُراجَع وتُعتمَد قبل الانتقال للتالية. هذا يوفر ٧٠٪ من الكريديت ويمنع الـ architecture drift الذي يدمر المشاريع.

الخريطة الكاملة للمحركات — ٣٠ محركاً

المادة ١٠.١

المحركات الـ١٦ الحضارية + ١٤ محركاً تجارياً

المحركات الـ١٦ في الوثيقة الأصلية كُتبت بمنظور حضاري فلسفي — سلسبيل في أقصى نضجها بعد ١٥+ سنة. أما المحركات التجارية الـ١٤ فهي محركات تشغيلية يومية تحتاجها ريف المدينة الآن. كلاهما إلزامي — لكن بترتيب مختلف.

المنتج كيان حي لا مجرد سجل

المنتج في سلسبيل ليس اسماً وسعراً في جدول. هو كيان حي يحمل DNA كاملاً يشمل الهوية والاقتصاد والسلوك والإمداد. هذا ما يجعل سلسبيل Commerce Intelligence وليس مجرد POS.

نظام شركاء النجاح — Partners & Success Network

المادة ١٣.١

شريك النجاح — مفوّض حضاري لا مندوب مبيعات

شريك النجاح ليس مندوب مبيعات يجلب عملاء ويختفي. هو مفوّض حضاري يقدم خدمة حقيقية، يتابع نجاح العميل، ويبني الثقة بالنيابة عن سلسبيل. الناس تثق فيمن تعرف — وهذا أقوى من كل إعلان.

المادة ١٣.٢

هيكل مستويات شركاء النجاح

المستوى ١ — عضو
Entry

مشاركة رابط، دعوة أصدقاء، مكافآت بسيطة. سمعة تبدأ من الصفر.

المستوى ٢ — شريك
Partner

يساعد العملاء، يحل المشاكل، يتابع الرضا. عمولة متكررة.

المستوى ٣ — مفوّض
Delegate

يدخل المتاجر والشركات، يُقنع بالانضمام، يبقى مسؤولاً.

المستوى ٤ — قائد منطقة
Leader

يدير شبكة مفوضين، هدف جماعي، تقارير حية.

المستوى ٥ — سفير مدينة
Ambassador

يمثل سلسبيل في مدينة كاملة. نفوذ + أدوات + دخل متميز.

المستوى ٦ — شريك سيادي
Sovereign

شريك استراتيجي. حصة في النمو الإقليمي. مقعد استشاري.

قوانين العمولة — إلزامية

  • العمولة مرتبطة بـ نشاط العميل الحقيقي لا بعدد المدعوّين
  • استمرار نشاط العميل = دخل متكرر للشريك
  • تقييم العميل لتجربته يؤثر مباشرة على عمولة الشريك
  • تحقيق Target الشهري = Bonus إضافي
  • انخفاض churn Rate = حوافز الاحتفاظ
  • كل عمولة تُسجَّل في Ledger Engine الموحد — لا نظام منفصل
  • عمولة على عدد الأشخاص المدعوّين فقط بدون قيمة حقيقية
  • هياكل هرمية يربح فيها الشريك من جهد من تحته (MLM محظور)
  • شريك يصل لبيانات العملاء الشخصية أو المالية
  • ترقية شريك بدون تقييم أداء موضوعي من حكيم
المادة ١٣.٣

نظام سمعة الشريك — Reputation Engine

كل شريك يمتلك ملف سمعة حي يشمل: تقييم العملاء، سرعة الاستجابة، معدل الاحتفاظ، عدد الشكاوى، كفاءة التدريب. إذا انخفضت السمعة: إنذار، ثم تجميد، ثم سحب العملاء منه وإعادة تعيينهم. النظام ينظف نفسه ذاتياً عبر حكيم.

Event Cortex — الجهاز العصبي الحي

المادة ١٤.١

Nothing Happens Silently — لا شيء يحدث بصمت

كل تغيير في سلسبيل يجب أن يصبح حدثاً حياً ينتشر عبر Event Bus. بدون Events: لا AI حقيقي، لا تعلم، لا توقع، لا Automation. Events هي الجهاز العصبي — بدونها النظام أصم وأعمى.

// ❌ خاطئ — تعديل صامت UPDATE products SET stock = 5 WHERE id = $1 UPDATE wallet SET balance = balance - 150 // ✓ صحيح — كل شيء Event emit('StockReduced', { product_id, qty, branch_id, actor }) emit('StockThresholdReached',{ product_id, level: 'low' }) emit('SaleCompleted', { order_id, items, cashier, branch, time }) emit('PriceChanged', { product_id, old_price, new_price, reason }) emit('PartnerCommissionDue', { partner_id, amount, trigger: 'customer_order' }) emit('LowStockDetected', { product_id, suggestion: 'reorder' }) // هيكل كل Event — إلزامي interface CivilizationEvent { event: string; // SaleCompleted, StockReduced... entity_id: string; actor_id: string; timestamp: string; context: Record<string, unknown>; payload: Record<string, unknown>; idempotency_key: string; }
المادة ١٤.٢

Event Multiplier — حدث واحد يحرّك المنظومة كلها

قوة الـ Event Architecture تظهر في التسلسل: حدث SaleCompleted واحد يُطلق: تحديث المخزون + تحديث توقعات الطلب + تحليل احتمالية الاحتيال + توليد توصيات المنتجات المرتبطة + حساب عمولة الشريك + تحديث Semantic Memory. كل هذا من event واحد.

قائمة Events الإلزامية — MVP

  • SaleCompleted / SaleCancelled / SaleRefunded
  • ProductAdded / ProductUpdated / ProductDeactivated
  • StockReduced / StockThresholdReached / StockRestocked
  • PriceChanged / DiscountApplied / PromotionTriggered
  • PartnerReferralConverted / PartnerCommissionDue / PartnerReputationChanged
  • UserRoleShifted / PermissionGranted / PermissionRevoked
  • FraudSuspected / AnomalyDetected / SystemAlert
  • SupplierOrderPlaced / DeliveryCompleted / InvoiceGenerated

Stem Cell Architecture — بنية الخلايا الجذعية

المادة ١٥.١

النواة الفارغة تنمو حسب البيئة

بدلاً من بناء نظام بقالة + نظام مطعم + نظام صيدلية، تُبنى نواة ذكية واحدة تكتشف نوع النشاط تلقائياً وتنمو حسبه. لا يُثبَّت نسخة جديدة لكل قطاع — نفس النواة تتشكل.

آلية الـ Morphing الإلزامية

  • النظام يكتشف نوع النشاط من capabilities المسجّلة — لا من if/else
  • الواجهة تتشكل Morph UI حسب: الجهاز + الدور + السياق + البيئة
  • الهاتف → أزرار ضخمة / الكمبيوتر → لوحة تحكم / التابلت → وضع لمس سريع
  • مطعم → يظهر تلقائياً: Kitchen Mode + Table Management + KDS
  • صيدلية → يظهر: Prescription + Medical Capability + Cold Chain
  • هايبر ماركت → يظهر: Shelf Intelligence + Scale Integration + Batch POS
  • Universal Printing: يتعرف على أي طابعة تلقائياً بدون تعريف يدوي
  • Smart Barcode: يلتقط أي Input Stream دون الحاجة للوقوف في خانة محددة
المادة ١٥.٢

Offline-First — يعمل بدون إنترنت دائماً

الكاشير لا يتوقف إذا انقطع الإنترنت. النظام يعمل Offline-First مع مزامنة ذكية عند عودة الاتصال. Conflict Resolution عبر Event Sourcing — لا بيانات تضيع.

الأمان — Zero Trust Architecture

المادة ١٧.١

لا ثقة بأحد — Zero Trust بلا استثناء

لا يُصدَّق على أي طلب من الـ client دون تحقق سيرفر. لا يُصدَّق على أي Role مُرسَل من الـ frontend. لا يُصدَّق على أي tenant_id غير موثّق من السيرفر.

قواعد الأمان الإلزامية

  • كل Supabase Edge Function تتحقق من الـ JWT قبل أي عملية
  • RLS مُفعَّل على كل جدول — لا جداول بدون Row Level Security
  • Service Role Key محظور في أي context مُعرَّض للويب
  • CSP مُفعَّل خصوصاً مع Dynamic UI
  • Rate Limiting على كل endpoint خصوصاً: AI, Auth, Payments, Partner APIs
  • كل secrets في Vault — لا في .env frontend ولا في logs
  • Signed Layouts — كل layout يحمل checksum ويُتحقق منه قبل الـ render
  • كل Event يحمل idempotency_key لمنع التكرار
  • Partner endpoints لها Rate Limiting مستقل لمنع Spam
